# Plan 05.1-01: Inbound Twilio Migration — env, HMAC, webhook handler, queue type

**Phase:** 05.1 — Twilio WhatsApp Migration
**Plan:** 1 of 2
**Blocking:** Server cannot start until `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, and `TWILIO_WHATSAPP_NUMBER` are in `validateEnv()` — currently only Meta vars are validated.

## Goal

Swap the inbound WhatsApp path from Meta Cloud API format to Twilio format:
replace the four Meta env vars with three Twilio vars; replace `verifyWhatsAppHmac` with `verifyTwilioSignature` (base64 HMAC over URL + sorted params); rewrite `POST /webhook/whatsapp` to parse Twilio's form-encoded body and strip the `whatsapp:` phone prefix; rename `waMessageId → messageSid` in `HeartbeatJobData`. The GET hub-verification handler is deleted — Twilio uses POST only.

## Context

**What already works (do not touch):**
- `src/routes/api.ts` — `handleConfirmSend` already sends via Twilio raw-fetch using `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_WHATSAPP_NUMBER`. Keep as-is.
- `POST /api/voice/playback` — already uses Twilio Basic-auth for media fetch. Keep as-is.
- `src/lib/phone.ts` — `normaliseE164` is correct; it is called AFTER stripping the `whatsapp:` prefix.
- `src/queue/worker.ts` — does NOT destructure `waMessageId` from job data; only field rename in the interface is needed.

**Twilio inbound payload format (form-encoded POST):**
```
From=whatsapp%3A%2B27821234567   ← sender, strip "whatsapp:" prefix
To=whatsapp%3A%2B14155238886     ← our number (TWILIO_WHATSAPP_NUMBER)
Body=Hello+there                  ← message text
MessageSid=SMxxxxxxxxxxxxxxxx      ← dedup key, replaces waMessageId
NumMedia=0                         ← 0 for text, 1+ for voice/image
MediaContentType0=audio/ogg        ← only present when NumMedia > 0
MediaUrl0=https://api.twilio.com/… ← only present when NumMedia > 0
```

**Twilio signature algorithm:**
1. Sort POST params alphabetically by key name (decoded, case-sensitive)
2. Concatenate each pair as `key + value` (no separator between pairs)
3. Prepend the full webhook URL: `url + concatenatedParams`
4. `HMAC-SHA256(data, TWILIO_AUTH_TOKEN)` → base64-encode
5. Compare with `X-Twilio-Signature` header (base64 string)

The raw body middleware already captures the form-encoded string in `c.get('rawBody')`. Parse it with `new URLSearchParams(rawBody)` after the signature check. **Never call `c.req.json()` — Twilio sends form data, not JSON.**

**No SDK needed:** `verifyTwilioSignature` is implemented manually with Node's `crypto` module. The `twilio` npm package is NOT required for inbound verification.

### Task 2: Replace `src/lib/hmac.ts` — Twilio signature verification

**File:** `src/lib/hmac.ts`
**Action:** Overwrite (replace the entire file — keep the same export shape, rename and reimplement)

The existing `verifyWhatsAppHmac` export is replaced with `verifyTwilioSignature`. All callers are updated in Task 3.

```typescript
// src/lib/hmac.ts
import crypto from 'node:crypto'

/**
 * Verify a Twilio webhook request signature.
 *
 * Algorithm:
 *   1. Sort POST params alphabetically by key (decoded values)
 *   2. Concatenate as key₁value₁key₂value₂… (no separator)
 *   3. Prepend the full webhook URL: url + concatenatedParams
 *   4. HMAC-SHA256(data, authToken) → base64
 *   5. Constant-time compare with X-Twilio-Signature header value
 *
 * @param url       - Full webhook URL including protocol and host (e.g. https://…/webhook/whatsapp)
 * @param params    - Decoded POST parameter key→value pairs (from URLSearchParams)
 * @param signature - Value of X-Twilio-Signature header (base64 string)
 * @param authToken - TWILIO_AUTH_TOKEN
 * @returns true if signature is valid, false otherwise
 */
export function verifyTwilioSignature(
  url: string,
  params: Record<string, string>,
  signature: string,
  authToken: string
): boolean {
  if (!signature) return false

  // Step 1 + 2: sort keys, concatenate key+value pairs
  const sortedKeys = Object.keys(params).sort()
  const paramStr = sortedKeys.map(k => `${k}${params[k]}`).join('')

  // Step 3: prepend URL
  const data = url + paramStr

  // Step 4: HMAC-SHA256, base64-encode
  const expected = crypto
    .createHmac('sha256', authToken)
    .update(data, 'utf8')
    .digest('base64')

  // Step 5: constant-time comparison (buffer lengths must match)
  try {
    const expectedBuf  = Buffer.from(expected, 'base64')
    const receivedBuf  = Buffer.from(signature, 'base64')
    if (expectedBuf.length !== receivedBuf.length) return false
    return crypto.timingSafeEqual(expectedBuf, receivedBuf)
  } catch {
    return false
  }
}
```

### Task 3: Rewrite `src/routes/webhook.ts` — Twilio inbound handler

**File:** `src/routes/webhook.ts`
**Action:** Overwrite (replace the entire file)

Key changes from Meta version:
- Remove `GET /whatsapp` hub-verification handler entirely
- Import `verifyTwilioSignature` (not `verifyWhatsAppHmac`)
- Parse form-encoded body (not JSON)
- Strip `whatsapp:` prefix from `From` field before calling `normaliseE164`
- Use `MessageSid` (not `message.id`) as the dedup/log identifier
- Use `process.env.TWILIO_WHATSAPP_NUMBER` for `to_phone` (not `+${WHATSAPP_PHONE_NUMBER_ID}`)
- Detect media via `NumMedia` field (not `message.type`)
- Rename local variable `waMessageId` → `messageSid` throughout

```typescript
// src/routes/webhook.ts
// Phase 05.1: Twilio WhatsApp inbound handler (replaces Meta Cloud API format)
// Raw body is available as c.get('rawBody') — set by middleware in server.ts
// Twilio sends form-encoded POST — never call c.req.json() here.
import { Hono } from 'hono'
import { supabase } from '../db/client'
import { normaliseE164 } from '../lib/phone'
import { verifyTwilioSignature } from '../lib/hmac'
import { enqueueHeartbeat } from '../queue/heartbeat'

export const webhookRouter = new Hono()

// POST /webhook/whatsapp — Twilio signature verification + message handler
// Twilio does NOT use a GET hub-verification step.
webhookRouter.post('/whatsapp', async (c) => {
  // STEP 1: TWILIO SIGNATURE VERIFICATION
  // rawBody is the raw application/x-www-form-urlencoded string captured by
  // the middleware in server.ts. Parse with URLSearchParams for both
  // signature validation and field extraction.
  const rawBody  = c.get('rawBody') as string
  const signature = c.req.header('X-Twilio-Signature') ?? ''

  // Parse decoded params for signature validation
  const params    = new URLSearchParams(rawBody)
  const paramsObj: Record<string, string> = {}
  params.forEach((v, k) => { paramsObj[k] = v })

  // Full URL is required for Twilio's HMAC — must match what Twilio sends to
  const url = c.req.url

  const signatureValid = verifyTwilioSignature(
    url,
    paramsObj,
    signature,
    process.env.TWILIO_AUTH_TOKEN!
  )

  if (!signatureValid) {
    console.warn('[Webhook] Twilio signature verification failed')
    return c.json({ error: 'Unauthorized' }, 401)
  }

  // STEP 2: EXTRACT FIELDS
  // Twilio form fields (decoded by URLSearchParams):
  const fromRaw   = params.get('From') ?? ''   // "whatsapp:+27821234567"
  const messageSid: string = params.get('MessageSid') ?? ''
  const messageBody: string | null = params.get('Body') || null

  // Strip the "whatsapp:" prefix before normalising to E.164
  const rawPhone  = fromRaw.replace(/^whatsapp:/, '')   // "+27821234567"

  if (!messageSid || !rawPhone) {
    return c.json({ received: true }, 200)
  }

  // STEP 3: DETECT MEDIA TYPE
  const numMedia  = parseInt(params.get('NumMedia') ?? '0', 10)
  const mediaType: string | null = numMedia > 0
    ? (params.get('MediaContentType0')?.split(';')[0].trim() ?? null)
    : null
  const mediaId: string | null = numMedia > 0
    ? (params.get('MediaUrl0') ?? null)
    : null

  // STEP 4: NORMALISE SENDER PHONE (ISO-02)
  const phone = normaliseE164(rawPhone)

  // STEP 5: UPSERT SENDER TO users (WA-03)
  const { data: userRow, error: upsertErr } = await supabase
    .from('users')
    .upsert({ phone }, { onConflict: 'phone' })
    .select('id')
    .single()

  if (upsertErr || !userRow) {
    console.error('[Webhook] User upsert failed:', upsertErr)
    return c.json({ error: 'DB error' }, 500)
  }

  const userId: string = userRow.id

  // STEP 6: PERSIST MESSAGE TO message_log (WA-05)
  // wa_message_id stores the Twilio MessageSid — column name unchanged in DB schema
  const { data: logRow, error: logErr } = await supabase
    .from('message_log')
    .insert({
      user_id:       userId,
      wa_message_id: messageSid,
      direction:     'in',
      from_phone:    phone,
      to_phone:      process.env.TWILIO_WHATSAPP_NUMBER!,
      body:          messageBody,
      media_type:    mediaType,
      media_id:      mediaId,
    })
    .select('id')
    .single()

  if (logErr || !logRow) {
    console.error('[Webhook] message_log insert failed:', logErr)
    return c.json({ error: 'DB error' }, 500)
  }

  const messageLogId: string = logRow.id

  // STEP 7: REDIS DEDUP GATE + BullMQ ENQUEUE
  // SET msg:{messageSid} 1 EX 7200 NX — prevents double-processing on Twilio retries
  const deduped = await enqueueHeartbeat({
    userId,
    messageLogId,
    messageSid,
    phone,
    messageBody,
    mediaType,
    mediaId,
  })

  if (!deduped) {
    console.warn(`[Webhook] Duplicate message ignored: ${messageSid}`)
    return c.json({ received: true }, 200)
  }

  // STEP 8: RETURN 200 IMMEDIATELY
  return c.json({ received: true }, 200)
})
```

## Notes

- `TWILIO_WHATSAPP_NUMBER` is stored WITH the `+` prefix (e.g. `+14155238886`). `to_phone` uses it directly — no prefix manipulation needed (unlike the old `+${WHATSAPP_PHONE_NUMBER_ID}` pattern).
- The `wa_message_id` Supabase column name is NOT renamed — it stores whatever message identifier the provider sends; now it will hold the Twilio `MessageSid`. Only the TypeScript variable name changes (`messageSid`).
- `c.req.url` in a production Hono server with a reverse proxy may return the internal URL. If Twilio signatures fail in production, check whether `X-Forwarded-Proto` / `X-Forwarded-Host` reconstruction is needed. For the sandbox/demo this is out of scope.
- `src/routes/api.ts` is NOT touched — `handleConfirmSend` already uses Twilio raw-fetch for outbound. Do not add the `twilio` SDK package.
- `src/queue/worker.ts` is NOT touched — it does not destructure `messageSid`/`waMessageId` from job data.
