# Plan 02-02: BullMQ + ioredis Setup

**Phase:** 2 — Webhook + Heartbeat
**Plan:** 2 of 3
**Requirements:** HB-01

## Goal

Stand up a single ioredis connection and a BullMQ Queue + Worker that are initialised at server startup, prove the worker processes a synthetic test job end-to-end before any agent code is wired, and expose `enqueueHeartbeat()` — the dedup-gated enqueue function called by the webhook handler.

## Context

Phase 1 delivered the Hono server skeleton (`src/server.ts`) with a clearly documented registration order. BullMQ and ioredis are already installed:
- `bullmq@5.71.1`
- `ioredis@5.10.1`

The env var `REDIS_URL` is validated at startup via `validateEnv()` and is available as `process.env.REDIS_URL`.

**Critical ioredis constraint (from architectural decisions):** BullMQ requires `{ maxRetriesPerRequest: null }` on the ioredis connection. Without this, BullMQ throws `"maxRetriesPerRequest must be null"` at queue initialisation time. **Never use `Bun.redis` for BullMQ** — Bun's built-in Redis client is not compatible with BullMQ's internal ioredis API surface.

The heartbeat Queue must be initialised and the Worker registered before the server begins accepting webhook traffic. The server startup in `server.ts` currently ends at line 79 with `console.log('Server running on port 3000')`. The queue + worker boot must happen at module load time so they are ready by the time Bun calls `fetch`.

Plan 02-01 imports `enqueueHeartbeat` from `src/queue/heartbeat.ts`. That function is defined here.

Plan 02-03 imports `HeartbeatJobData` (the job payload type) and the worker processor from this plan's files.

## Tasks

### Task 1: Create ioredis singleton + BullMQ Queue + enqueueHeartbeat

**File:** `src/queue/heartbeat.ts`
**Action:** Create

Create the full heartbeat queue module. This file owns the ioredis connection, the BullMQ Queue, and the `enqueueHeartbeat()` function that the webhook handler calls.

```typescript
// src/queue/heartbeat.ts
// BullMQ heartbeat queue + ioredis connection.
//
// CONSTRAINT: IORedis must be constructed with { maxRetriesPerRequest: null }
// or BullMQ will throw at startup. Never use Bun.redis here.
//
// enqueueHeartbeat() is called by the webhook handler (Plan 02-01).
// The Worker processor is registered separately in src/queue/worker.ts (Plan 02-03)
// to keep this file importable in tests without spawning real workers.

import { Queue } from 'bullmq'
import IORedis from 'ioredis'

// Singleton ioredis connection shared by Queue, Worker, and dedup key operations.
// maxRetriesPerRequest: null is REQUIRED by BullMQ — do not remove.
export const redis = new IORedis(process.env.REDIS_URL!, {
  maxRetriesPerRequest: null,
})

redis.on('connect', () => console.log('[Redis] Connected'))
redis.on('error', (err) => console.error('[Redis] Error:', err))

// The heartbeat queue — jobs flow from webhook → queue → worker (Plan 02-03)
export const heartbeatQueue = new Queue('heartbeat', { connection: redis })

/**
 * Payload carried by every heartbeat BullMQ job.
 * Defined here so Plan 02-01 (webhook) and Plan 02-03 (worker) share the same type.
 */
export interface HeartbeatJobData {
  userId:        string   // UUID from users table
  messageLogId:  string   // UUID from message_log table
  waMessageId:   string   // WhatsApp message ID (used as dedup key)
  phone:         string   // E.164 sender phone
  messageBody:   string | null
  mediaType:     string | null
  mediaId:       string | null
}

/**
 * Enqueue a heartbeat job with Redis dedup protection.
 *
 * Dedup key: SET msg:{waMessageId} 1 EX 7200 NX
 *   - EX 7200 = 2-hour TTL (WhatsApp retries within this window)
 *   - NX      = only set if NOT exists → returns 'OK' on first call, null on duplicate
 *
 * @returns true  — job was enqueued (first occurrence)
 * @returns false — duplicate; job was NOT enqueued
 */
export async function enqueueHeartbeat(data: HeartbeatJobData): Promise<boolean> {
  const dedupKey = `msg:${data.waMessageId}`
  const result = await redis.set(dedupKey, '1', 'EX', 7200, 'NX')

  if (result === null) {
    // Key already existed — this is a duplicate delivery from WhatsApp
    return false
  }

  await heartbeatQueue.add('process', data, {
    attempts: 1,
    // timeout: 15000 ms — if worker takes longer than 15s, job is marked failed
    // BullMQ v4+ uses 'timeout' at job level via removeOnComplete/removeOnFail opts;
    // for v5 the per-job timeout is set here and honoured by the Worker
    removeOnComplete: { count: 100 },
    removeOnFail:     { count: 50 },
  })

  return true
}
```

**Note on job timeout:** BullMQ v5 does not accept `timeout` directly in `add()` options at the Queue level. The Worker enforces timeouts via its `opts.runRetryDelay` and the job's `opts` field when the job is added. To apply a 15-second timeout per job, pass it as part of the job options object in `add()`:

```typescript
await heartbeatQueue.add('process', data, {
  attempts: 1,
  timeout:  15_000,  // ms — worker aborts if processor takes longer
  removeOnComplete: { count: 100 },
  removeOnFail:     { count: 50 },
})
```

Confirm the installed BullMQ version's `JobsOptions` type accepts `timeout`. If it does not (older minor), wrap the job data in a try-catch and log the timeout separately in the Worker's `opts`.

### Task 2: Register Worker at server startup and prove with a synthetic test job

**File:** `src/queue/worker.ts`
**Action:** Create

Create the worker registration module. This is imported once in `src/server.ts` (after route setup) to boot the worker process. The worker processor is a placeholder stub that will be replaced in Plan 02-03 with the full surface decision gate logic.

```typescript
// src/queue/worker.ts
// BullMQ Worker for heartbeat jobs.
// Imported once at server startup (in server.ts) to begin processing.
//
// PHASE 2 STUB: processor logs the job and marks it complete.
// Plan 02-03 replaces processHeartbeat with the full surface decision gate.

import { Worker } from 'bullmq'
import { redis, type HeartbeatJobData } from './heartbeat'

/**
 * Heartbeat job processor — stub for Phase 2 wiring validation.
 * Plan 02-03 replaces this function body with the real gate logic.
 */
export async function processHeartbeat(job: { data: HeartbeatJobData }): Promise<void> {
  console.log(`[Worker] Processing job ${job.data.waMessageId} for user ${job.data.userId}`)
  // Stub: no-op. Plan 02-03 implements the surface decision gate here.
}

export const heartbeatWorker = new Worker<HeartbeatJobData>(
  'heartbeat',
  processHeartbeat,
  {
    connection: redis,
    concurrency: 5,
  }
)

heartbeatWorker.on('completed', (job) => {
  console.log(`[Worker] Job completed: ${job.id}`)
})

heartbeatWorker.on('failed', (job, err) => {
  console.error(`[Worker] Job failed: ${job?.id}`, err)
})
```

Add the following import immediately after the existing route imports (before the `app` declaration). Position matters — must be after `validateEnv()` runs but before the server export:

Add it on the line after `import { wsConnections } from './ws/manager'` and before `const app = new Hono()`.

## Notes

- Import order in `server.ts` is critical. `import './queue/worker'` must come AFTER `validateEnv()` runs (which it does — `validateEnv()` is called synchronously at line 14 before any other imports are resolved in the module graph). The import itself is a side-effect import — it instantiates the `Worker` object which opens the Redis connection immediately.
- The `redis` singleton is exported from `heartbeat.ts` and shared by both the Queue (heartbeat.ts) and the Worker (worker.ts). BullMQ requires Queue and Worker to share the same ioredis connection config (both with `maxRetriesPerRequest: null`). Using a shared export achieves this.
- When `REDIS_URL` is an Upstash URL (uses `rediss://` with TLS), ioredis handles TLS automatically via the URL scheme. No extra TLS config needed.
- `removeOnComplete: { count: 100 }` keeps only the last 100 completed jobs in Redis — prevents memory growth for a long-running hackathon demo server.
- `heartbeatWorker.concurrency: 5` allows 5 parallel job processors. For a hackathon with a single demo user, 1 would suffice — but 5 is safe and avoids queue backup if multiple messages arrive in a burst.
- Do not import `heartbeat.ts` from test files that run without a real Redis connection — the IORedis constructor connects immediately on import. Use dependency injection or mock the module in tests.
