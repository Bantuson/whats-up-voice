<objective>
Fix the Bun 1.3.x mock contamination that causes 10 failures in whatsapp.test.ts when the full
test suite runs after voiceCommand.test.ts.

Root cause: In Bun 1.3.x, `mock.module()` calls are persistent within the test process. Both
`voiceCommand.test.ts` and `whatsapp.test.ts` call `mock.module('../src/db/client', ...)`. The
voiceCommand mock factory returns `{ supabase: { from: mock(() => ({ insert: mockInsert })) } }` —
a shallow mock with only an `insert` method. When whatsapp.test.ts runs in the same process after
voiceCommand.test.ts, the module mock is already set to the voiceCommand factory. The whatsapp
`mockFrom` reference still exists, but calling `mockFrom.mockImplementation(...)` in tests does
not affect the supabase singleton that was already evaluated against the voiceCommand factory.

Fix strategy: In `whatsapp.test.ts`, add a `beforeEach` at the top level (before any describe
blocks) that calls `mockFrom.mockReset()` and sets a safe default implementation that returns
the correct chainable mock structure. This ensures each test starts with a known-good mockFrom
state regardless of what ran before it in the process.

This is a test harness fix. No production code changes.

Purpose: Make `bun test` (full suite) exit 0 — closing the partial gap on truth #31.
Output: Updated tests/whatsapp.test.ts with cross-file-safe mock reset in beforeEach.
</objective>

<!-- Bun 1.3.x mock contamination pattern and fix -->

The contamination:
- voiceCommand.test.ts: `mock.module('../src/db/client', () => ({ supabase: { from: mock(() => ({ insert: mockInsert })) } }))`
- whatsapp.test.ts: `mock.module('../src/db/client', () => ({ supabase: { from: mockFrom } }))`
- When voiceCommand runs first, `supabase.from()` returns `{ insert: mockInsert }` at module evaluation time
- whatsapp.test.ts calls `mockFrom.mockImplementation(...)` but the module was already bound to voiceCommand's factory
- Result: 10 failures because mockFrom implementations (chainable query builders) are never applied

The fix in whatsapp.test.ts:
```typescript
// Add this BEFORE any describe blocks, AFTER mock.module declarations
beforeEach(() => {
  // Reset mockFrom to a safe default so voiceCommand mock contamination
  // does not affect this test file (Bun 1.3.x mock.module is process-persistent)
  mockFrom.mockReset()
  mockFrom.mockImplementation(() => ({
    select: () => ({
      eq: () => ({
        eq: () => ({
          order: () => ({
            limit: () => Promise.resolve({ data: [], error: null }),
          }),
          ilike: () => ({
            single: () => Promise.resolve({ data: null, error: null }),
          }),
          single: () => Promise.resolve({ data: null, error: null }),
        }),
      }),
    }),
  }))
})
```

This default is a safe "empty result" chainable structure. Individual tests then override with
`mockFrom.mockImplementation(...)` inside each test body (existing pattern — no test body changes
needed). The key is establishing a non-crashing default at the top of each test so tests that
do not override mockFrom still get a valid chain.
